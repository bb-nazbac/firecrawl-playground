Qualifying Agentic System

A high-throughput, spec-driven pipeline for qualifying business leads through intelligent web scraping and LLM-based analysis. Processes thousands of domains with configurable classification rules.

System Overview

The Qualifying Agentic System (QAS) is a production-grade pipeline that analyzes company websites to determine if they meet specific qualification criteria. It uses a layered approach combining web scraping (Firecrawl) with LLM analysis (OpenAI GPT models) to extract structured data and make classification decisions.

L1: Homepage Analysis

The first layer scrapes and analyzes the company's homepage to make an initial qualification decision. This is the most cost-effective stage, as many domains can be classified with just homepage content.

Process Flow

1. Homepage Scrape
Firecrawl fetches the homepage HTML, converts to clean markdown. Handles redirects, JavaScript rendering, and rate limiting.
2. Waterfall Filter (Optional)
Quick pre-check using homepage content to disqualify obvious non-fits (competitors, wrong TLDs, blocklisted patterns). Saves LLM costs.
3. LLM Analysis
GPT analyzes the homepage markdown against spec questions. Extracts answers with confidence levels and evidence.
4. Sufficiency Check
Evaluates if enough critical questions are answered with sufficient confidence. If yes, skip L2 and classify immediately.

Sufficiency Criteria

A homepage is considered sufficient when:

  • Minimum questions answered: Typically 4 of the critical questions
  • All required questions answered: Questions marked as required in spec must have answers
  • Confidence threshold met: Answers must have at least "medium" confidence
Cost Optimization
In typical runs, 15-20% of domains are fully classified at L1 (homepage only), saving significant L2 costs. The waterfall filter can disqualify an additional 5-10% before any LLM analysis.

L2: Map + Iterate

When the homepage doesn't provide sufficient information, L2 discovers and analyzes additional pages through an intelligent iterative process.

Map Phase

Uses Firecrawl's /map endpoint to discover all accessible pages on the domain:

# Firecrawl /map response example
{
    "links": [
        "https://example.com/about",
        "https://example.com/products",
        "https://example.com/services",
        "https://example.com/contact",
        "https://example.com/careers",
        ...
    ]
}

Page Selection Strategy

The LLM selects pages based on:

  • Relevance to unanswered questions: Prioritizes pages likely to contain missing information
  • Page priority hints: Spec can define preferred pages (about, products, services)
  • URL patterns: Recognizes common patterns like /about-us, /what-we-do

Waterfall Filter

The waterfall filter is an optional pre-qualification stage that runs before the main LLM analysis. It uses simple pattern matching and a quick LLM check to disqualify obvious non-fits, saving significant costs.

Filter Components

Filter Type Description Example
Domain Blocklist Known competitor domains to skip entirely accenture.com, deloitte.com, ibm.com
TLD Skip Skip domains with certain TLDs .gov, .edu
Content Patterns Disqualify if homepage contains certain strings "sap consulting", "erp implementation"
Quick LLM Check Fast classification into broad categories PRODUCTION_COMPANY, IT_SERVICES_COMPETITOR

Spec Configuration

"waterfall_filter": {
    "enabled": true,
    "domain_blocklist": ["accenture.com", "deloitte.com"],
    "skip_domains_matching": [".gov", ".edu"],
    "disqualify_if_homepage_contains": [
        "sap consulting",
        "erp implementation",
        "systems integrator"
    ],
    "questions": [{
        "field": "quick_business_check",
        "prompt": "What type of business is this?",
        "answer_options": ["PRODUCTION_COMPANY", "IT_SERVICES_COMPETITOR"]
    }],
    "disqualify_rules": [{
        "if_field": "quick_business_check",
        "equals": "IT_SERVICES_COMPETITOR",
        "then_disqualify": true,
        "reason": "COMPETITOR"
    }]
}

Sufficiency Check

The sufficiency check determines whether enough information has been gathered to make a confident classification decision. This is the key mechanism for early termination and cost optimization.

Configuration

"sufficiency_check": {
    "enabled": true,
    "min_questions_answered": 4,
    "required_questions": [
        "is_production_oriented",
        "us_operations",
        "is_competitor",
        "industry"
    ],
    "min_confidence": "medium"
}

Evaluation Logic

  1. Count how many questions have non-null answers
  2. Check that all required_questions are answered
  3. Verify each answer meets the min_confidence threshold
  4. If all criteria met → sufficient, proceed to classification
  5. If not met → continue to next iteration or L2

Spec System

Specifications are JSON configuration files that define the entire qualification logic. They are the single source of truth for what questions to ask, how to classify, and what outputs to produce.

Spec Locations

configs/specs/analysis/
├── example_sap_qualification.json      # Example - SAP manufacturing
├── example_duty_drawback_qualification.json  # Example - duty drawback
└── [client]_[use_case].json        # Pattern for new specs

Key Benefits

  • No code changes: Modify qualification criteria by editing JSON only
  • Version control: Track changes to business rules over time
  • Client isolation: Each client has their own spec file
  • Reproducibility: Same spec always produces same behavior

Spec Structure

A complete spec file contains the following sections:

1. Metadata

{
    "spec_name": "example_sap_qualification",
    "description": "SAP qualification for manufacturing companies",
    "version": "1.4",
    "critical_questions": ["is_production_oriented", "us_operations"]
}

2. Categories

"categories": [
    {
        "name": "MANUFACTURING",
        "description": "Companies that manufacture physical products"
    },
    {
        "name": "FOOD_AND_BEVERAGE",
        "description": "Food and beverage manufacturers"
    },
    {
        "name": "DISQUALIFIED",
        "description": "Does not meet qualification criteria"
    }
]

3. Questions

"questions": [
    {
        "field": "is_production_oriented",
        "question": "Is this an industrial-scale manufacturer?",
        "answer_type": "boolean",
        "reasoning_required": true,
        "evidence_required": true,
        "guidance": "YES only if they own factories that produce goods at scale..."
    },
    {
        "field": "industry",
        "question": "What industry is this company in?",
        "answer_type": "string",
        "answer_options": ["Manufacturing", "Food & Beverage", "Other"]
    }
]

4. Classification Logic

"classification_logic": {
    "apply_in_order": [
        {
            "order": 1,
            "condition": "is_competitor = true",
            "result": "DISQUALIFIED",
            "disqualification_reason": "competitor"
        },
        {
            "order": 2,
            "condition": "is_production_oriented = false",
            "result": "DISQUALIFIED",
            "disqualification_reason": "not_production_oriented"
        },
        {
            "order": 5,
            "condition": "industry = 'Manufacturing'",
            "result": "MANUFACTURING"
        }
    ]
}

5. Examples (for LLM context)

"examples": {
    "qualified": [
        {"domain": "tysonfoods.com", "classification": "FOOD_AND_BEVERAGE"}
    ],
    "disqualified": [
        {"domain": "accenture.com", "reason": "competitor"}
    ]
}

Classification Logic

Classification rules are evaluated in order. The first matching rule determines the final classification. This allows for a clear priority hierarchy.

Condition Syntax

Syntax Description Example
field = value Exact match (case-insensitive) is_competitor = true
field = 'value' String match with quotes industry = 'Manufacturing'
field in [a, b, c] Value is one of multiple options tier in [HIGH, MEDIUM]
default Catch-all fallback rule Always matches if reached
Important: Rule Order Matters
Rules are evaluated top-to-bottom by order field. Place disqualification rules first to catch invalid leads before category assignment.

Usage Guide

Basic Command

python -m core_openai.pipeline <input.csv> \
    --spec <path/to/spec.json> \
    --client <client_name> \
    --run-name <run_identifier>

All Options

Option Default Description
--spec required Path to spec JSON file
--client required Client identifier (used in output path)
--run-name timestamp Run identifier for output folder
--model gpt-5-mini OpenAI model to use
--firecrawl-concurrency 50 Max parallel Firecrawl requests
--openai-concurrency 150 Max parallel OpenAI requests
--min-pages 5 (deep) Minimum pages to scrape (deep only)
--max-pages 15 (deep) Maximum pages to scrape (deep only)
--no-waterfall false Skip waterfall filter

Path Values

Path Meaning
filtered_early Disqualified by waterfall filter (lowest cost)
homepage_sufficient Classified using only homepage (low cost)
homepage_plus_iterate Required additional pages (higher cost)
failed Domain unreachable or scrape error

Cost Optimization Tips
  • Use waterfall filters to eliminate obvious non-fits early
  • Tune sufficiency criteria to terminate earlier when possible
  • Use gpt-5-nano for lower-stakes classifications
  • Reduce max_pages in deep pipeline if full analysis isn't needed

Model Selection

Available Models

Model Speed Cost Best For
gpt-5-mini Fast Medium Production runs, balanced quality/cost
gpt-5-nano Very Fast Low High-volume screening, simple criteria
gpt-4o Medium High Complex reasoning, nuanced classification

Model Selection Guidelines

  • Default to gpt-5-mini for most production workloads
  • Use gpt-5-nano when qualification criteria are simple and clear-cut
  • Use gpt-4o for complex multi-factor decisions or when accuracy is critical
